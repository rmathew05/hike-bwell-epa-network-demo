The Problem

The rule that should streamline care will stall at adoption

  • !
    1,000+ endpoints, no registry. Each provider must independently discover, credential, and integrate every payer API — a multi-year effort for large systems.
  • !
    EHR fragmentation. Legacy/niche EHRs, DME suppliers, and ambulatory practices can't natively consume FHIR PA APIs.
  • !
    No provider mandate. Without a frictionless on-ramp, the default stays fax and portals — repeating the underused Patient Access API.
The Solution

A national ePA network — the HIE playbook, applied to prior auth

  • ✓
    Single FHIR endpoint. Integrate once via API, a SMART on FHIR app in-EHR, or a standalone portal.
  • ✓
    Non-compatible EHR coverage. Patent-pending methods bring legacy / non-FHIR EHRs into electronic PA without migration.
  • ✓
    Bidirectional routing. Routes requests to the right payer and returns decisions in real time — aligned to Da Vinci CRD, DTR & PAS.
  • ✓
    Pre-seeded payer connections. Existing health-system & DME relationships accelerate time-to-value.

End-to-End · Da Vinci CRD → DTR → PAS

Live ePA request

Watch a prior authorization travel from the provider's EHR, through the HIKE hub, to the payer and back — at each Da Vinci stage. Payloads are illustrative FHIR.

CRDDTRPAS
1

Provider submits order EHR → HIKE

Clinician orders MRI Lumbar Spine. SMART on FHIR app packages the request and sends it to the single HIKE endpoint.
POST /hike/fhir/ServiceRequest "resourceType": "ServiceRequest", "code": "72148" // MRI lumbar w/o contrast, "subject": "Patient/maria-gonzalez", "insurer": "UnitedHealthcare-MA"
2

HIKE resolves payer & queries CRD Coverage Requirements Discovery

Hub looks up the member's payer in the network registry and calls that payer's CRD service. Is prior auth required?
GET /uhc/crd/coverage-requirements?code=72148 → { "priorAuthRequired": true, "dtrTemplate": "imaging-lumbar-v3" }
3

DTR auto-fills clinical documentation Documentation Templates & Rules

The payer's questionnaire is fetched and pre-populated from the EHR chart — conservative therapy tried, duration, prior imaging — with no manual re-keying.
QuestionnaireResponse: imaging-lumbar-v3 "conservativeTherapy": "6 weeks PT + NSAIDs" (auto), "redFlags": "progressive neuro deficit" (auto), "priorImaging": "none in 12 mo" (auto)
4

HIKE submits PAS to payer Prior Authorization Support

Bundled request routed to the payer's PAS endpoint. HIKE handles transport, auth, and format translation — including for non-FHIR-native providers.
POST /uhc/pas/Claim/$submit "use": "preauthorization", "item": "72148", "supportingInfo": "QuestionnaireResponse" → routing… awaiting payer decision
5

Decision returns to the provider HIKE → EHR

Payer adjudicates and returns the decision through the hub to the originating EHR — status, auth number, and rationale, in real time.
ClaimResponse "outcome": "complete", "disposition": "Approved", "preAuthRef": "AUTH-2027-77310"
